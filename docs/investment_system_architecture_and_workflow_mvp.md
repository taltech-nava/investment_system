# Investment Prediction & Learning System — Student Project MVP
## Architecture Document

---

## 1. Purpose and Core Logic

The system exists to answer one question over time: **which sources and methods produce accurate investment price predictions, and which do not?**

To answer it, the system must:
- Record price predictions before outcomes are known, with full provenance (source, method, conviction)
- Record the realised price at the predicted maturation date automatically
- Compare forecast to realised systematically
- Accumulate enough forecast–realised pairs to detect patterns across sources, methods, horizons, and asset classes

There are no shortcuts. Backtesting retrofits strategy to known outcomes. Retrospective recall is distorted by hindsight bias. The only valid data is predictions made before the fact, locked at creation time, compared to reality at maturation time.

Automation is not a convenience — it is a requirement for the same reason. If the pipeline does not run consistently and identically every time, the resulting database will reflect human selectivity and attention bias rather than systematic prediction behavior. A manually curated database will never accumulate the clean, comparable observations needed for pattern learning. The system must run the same procedure on every source, every time, regardless of what the user already suspects the outcome to be.

**Price prediction accuracy is the sole outcome variable in the MVP.** Thesis generation and thesis comparison are deferred — not because they lack value, but because price prediction is what matters for investment decisions, and thesis correctness does not carry over to price correctness since the two tasks use different sources and methods.

## 2. System Modules

### 2.1 Search & Fetch Module *(exists)*
Searches the internet using structured Google dork queries — keywords, tickers, `site:` domain constraints, `filetype:` and other operators constructed programmatically per instrument and horizon. Uses Serper API for Google search. Downloads reports, articles, and data files into a temporary raw data folder. Stores all search metadata alongside each downloaded file: full keyword string, dork domain component, search engine, horizon context, fetch date.

### 2.2 Audit Module *(exists)*
Examines each file in the raw data folder. Assigns a quality rank (1–5) via LLM API call. Extracts source metadata (title, institution, authors, publication date, URL) either programmatically or via LLM. Writes every source to the `sources` table with a `status` field:
- `accepted` — quality rank 3–5, available to the Extraction & Synthesis module
- `rejected` — quality rank 1–2, retained for record but excluded from synthesis
- `failed` — scraping failure, paywall, or download error; quality rank is NULL

The audit runs the same procedure on every report regardless of what the user expects. This is not a convenience — manual curation reintroduces the selective attention and post-hoc rationalization the system is designed to eliminate.

### 2.3 Extraction & Synthesis Module *(to be built)*
Fetches accepted sources from the `sources` table (filtered by `instrument_id`, `horizon_context`, and `status = accepted`). Extracts text passages relevant to price prediction for the target instrument and horizon — via LLM API call. Collates extracted passages into a single text object. Constructs a prompt and sends it to an LLM. Receives a price prediction for the specified time horizon. Writes the prediction to `forecast_prices`. Writes the list of `source_id` values used into the `forecast_sources` bridge table. Source linkage is programmatic — the module must return and persist the source list at write time. This linkage is the structural spine of the pattern-learning module and must not be reconstructed after the fact.

### 2.4 Forecast Entry Module *(to be built)*
Handles manually entered price predictions — hunches, quick directional calls, predictions not generated by the pipeline. User fills a structured form: price target, currency, direction, conviction, method (self-declared), maturation date, horizon label. `entry_mode = manual` marks the row. No source entries in the bridge table for manual forecasts — the absence of source linkage is itself informative when analyzing accuracy by entry mode.

### 2.5 Realised Data Collection Module *(to be built)*
Triggered by the scheduler when a maturation date is reached. Fetches the closing price for the instrument from yfinance or equivalent data provider. Writes the result to `realised_prices`. No source linkage — the realised price is a market fact, not synthesized from research sources.

### 2.6 Scheduler *(to be built)*
Monitors `forecast_prices` for rows whose `maturation_date` equals or has passed today. Triggers the Realised Data Collection Module for each due row. Implemented with APScheduler or a daily cron job.

### 2.7 Comparison & Analytics Module *(to be built)*
Joins `forecast_prices` to `realised_prices` on `instrument_id` + `maturation_date`. Computes:

- **Price return error** (primary): `(actual_price − predicted_price) / predicted_price`
- **Direction correctness**: whether the predicted direction (long / short) matched the actual price movement
- Writes results to `divergence_reports`

**Pattern analysis** (after sufficient observations — approximately 30–50 completed pairs): joins `divergence_reports` through `forecast_sources` to `sources` and groups by `institution`, `source_type`, `method`, `quality_rank`, `conviction`, `horizon_label`, `asset_class`. This reveals which combinations of sources and methods produce the most and least accurate predictions — the core learning output of the system.

### 2.8 Interface *(to be built)*
Local web app (FastAPI + React + MySQL). Four screens:

- **Dashboard** — active forecasts, upcoming maturation dates, forecasts due for review today
- **Forecast workspace** — manual price prediction entry form
- **Review screen** — displays original forecast alongside realised price fetched automatically; user adds a free-text divergence note; comparison engine output displayed
- **Analytics view** — price accuracy distributions and groupby breakdowns by source / institution / method / conviction / horizon; meaningful after ~30–50 completed pairs

## 3. Workflow

```
FORECAST PIPELINE
─────────────────
[user_id + instrument_id + horizon_label + Google Dork Parameters]
  search_subjects, search_intents, search_filetypes,
  search_title_block, search_org_block,
  search_date_after, search_date_before
        │
        ▼
[Search & Fetch Module]
  → construct full dork query → store as search_query_full
  → send to Serper API (search_engine = google-serper)
  → raw files → temporary raw data folder
        │
        ▼
[Audit Module]
  → LLM assigns quality_rank 1-5 per file
  → LLM or programmatic extraction of metadata per file:
      title, institution, author_1/2/3,
      publication_date, url, source_type, method
  → ALL sources written to sources table with:
      user_id          ← carried from pipeline entry point
      instrument_id    ← carried from pipeline entry point
      horizon_label    ← carried from pipeline entry point
      mode = forecast
      fetch_date = today
      status = accepted   (rank 3-5)  ← available for synthesis
      status = rejected   (rank 1-2)  ← retained, excluded from synthesis
      status = failed     (scraping failure / paywall / download block)
        rejection_reason = scraping-failure / download-failure / other
        quality_rank = NULL
        file_path = NULL
        title = NULL
        │
        ▼
[Extraction & Synthesis Module]
  → fetch sources WHERE status = accepted
    AND instrument_id = [target] AND horizon_label = [target]
  → extract price-relevant text passages (LLM)
  → collate into single text object
  → send to LLM with price prediction prompt
  → receive predicted_price for target horizon
  → write to forecast_prices:
      user_id, instrument_id, creation_date,
      maturation_date, horizon_label, predicted_price,
      currency, conviction, method, entry_mode = pipeline
  → write to forecast_sources bridge:
      forecast_price_id, source_id (one row per source used)
      ← non-negotiable: must be written in same operation
        │
        ▼
[forecast_prices]
  ← also accepts manual entry via Forecast Entry Module:
     user_id, instrument_id, predicted_price, currency,
     conviction, method (user-declared), maturation_date,
     horizon_label, entry_mode = manual
     (no rows written to forecast_sources for manual entry)
        │
        ▼
[Scheduler reads forecast_prices.maturation_date]


REALISED PIPELINE  (triggered on maturation_date)
──────────────────
[Scheduler fires for each due forecast_prices row]
        │
        ▼
[Realised Data Collection Module]
  → fetch closing price from yfinance (data_source = yfinance)
  → write to realised_prices:
      instrument_id, creation_date = today,
      actual_price, currency, data_source
  → no user_id (system-generated, not user-initiated)
  → no source linkage (market fact, not synthesized)


COMPARISON & LEARNING
──────────────────────
[forecast_prices]
        │ JOIN on instrument_id + maturation_date ≈ realised.creation_date
[realised_prices]
        │
        ▼
[Comparison Engine]
  → price_return_error = (actual_price - predicted_price) / predicted_price
  → direction_correct = boolean (nullable)
  → write to divergence_reports:
      user_id          ← carried from forecast_prices.user_id
      forecast_price_id, realised_price_id,
      review_date = today, price_return_error,
      direction_correct, divergence_notes (user-written)
        │
        ▼
[Analytics Module]  (after ~30-50 completed pairs)
  divergence_reports
        │ JOIN forecast_sources ON forecast_price_id
  forecast_sources
        │ JOIN sources ON source_id
  sources
        │
        ▼
  GROUP BY institution / source_type / method /
           quality_rank / conviction / horizon_label /
           asset_class / entry_mode
        │
        ▼
  → identify high-accuracy and low-accuracy
    source + method combinations
```

## 4. Key Design Principles

**Price prediction accuracy is the sole outcome variable.** Thesis generation and comparison are deferred to a future version. The MVP focuses exclusively on the price prediction loop: forecast → realised → divergence → pattern.

**Automation is an anti-bias requirement, not a convenience.** The pipeline runs the same procedure on every source, every time, blind to outcome. Manual curation reintroduces selective attention and hindsight bias — the exact distortions the system is designed to eliminate. Consistent automation is what makes the accumulated database clean enough to learn from.

**Long format throughout.** One observation per row. All horizons in the same table, distinguished by `maturation_date` and `horizon_label`. Pivoting to wide is a reporting step, not a storage decision.

**One sources table for everything.** Accepted, rejected, and failed sources all live in `sources`, distinguished by `status`. Rejected and failed sources are retained — they record what was searched for and what was not usable plus the background/context information, which is its own signal over time.

**The bridge table is the learning engine.** `forecast_sources` is what makes pattern analysis possible. Without it, accuracy statistics cannot be attributed to sources or methods, and the system's core purpose is unreachable.

**The system requires time to become useful.** Pattern analysis becomes meaningful at approximately 30–50 completed forecast–realised pairs. Before that threshold the system produces individual divergence reports, which are useful but not yet pattern-revealing. This is not a design flaw — it is the price of building on real, unbiased data rather than retrofitted backtests or reconstructed memory.
